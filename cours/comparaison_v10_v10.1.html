SUPPRIME en v10.1 Section VI.3 - Algorithm requirements (v10.0 uniquement)

Toute cette section n'existe pas dans v10.1.

SUPPRIME VI.3.1 - Complexity model and operation constraints

La v10.0 imposait que la complexite soit mesuree en nombre d'operations push_swap generees, pas en complexite theorique classique.

SUPPRIME VI.3.2 - Disorder metric (mandatory)

La v10.0 imposait le calcul du disorder (nombre entre 0 et 1) avec ce pseudocode :

function compute_disorder(stack a):
    mistakes = 0
    total_pairs = 0
    for i from 0 to size(a)-1:
        for j from i+1 to size(a)-1:
            total_pairs += 1
            if a[i] > a[j]:
                mistakes += 1
    return mistakes / total_pairs

Regle : "You must measure the disorder before doing any moves."

v10.1 : Aucune mention du disorder.

SUPPRIME VI.3.3 - Required strategies (4 algorithmes obligatoires)

1. Simple algorithm - O(n2)

Exemples : insertion sort, selection sort, bubble sort, min/max extraction

2. Medium algorithm - O(n * sqrt(n))

Exemples : chunk-based sorting, block-based partitioning, bucket sort avec sqrt(n) buckets, range-based

3. Complex algorithm - O(n log n)

Exemples : radix sort (LSD/MSD), merge sort, quick sort, heap sort, binary indexed tree

4. Custom adaptive algorithm (basee sur le disorder)

Devait selectionner automatiquement une methode selon le disorder mesure :

RegimeConditionComplexite requise
Low disorderdisorder < 0.2O(n)
Medium disorder0.2 ≤ disorder < 0.5O(n * sqrt(n))
High disorderdisorder ≥ 0.5O(n log n)

Le README devait documenter les seuils, les techniques internes, et l'argument de complexite.

v10.1 : Aucune de ces exigences. Tu es libre de choisir ton/tes algorithme(s). Seuls les benchmarks comptent.

Le programme push_swap

SUPPRIME en v10.1 Strategy selector (flags)

La v10.0 imposait les flags suivants :

FlagEffet
--simpleForce l'algorithme O(n2)
--mediumForce l'algorithme O(n * sqrt(n))
--complexForce l'algorithme O(n log n)

v10.1 : Pas de flags. Juste ./push_swap [liste d'entiers]

SUPPRIME en v10.1 Mode benchmark (--bench)

La v10.0 imposait un mode --bench affichant sur stderr :

  • Le disorder calcule (% avec 2 decimales)
  • Le nom de la strategie et sa classe de complexite
  • Le nombre total d'operations
  • Le decompte de chaque operation (sa, sb, ss, pa, pb, ra, rb, rr, rra, rrb, rrr)

Exemple v10.0 :

[bench] disorder: 49.93%
[bench] strategy: Adaptive / O(n*sqrt(n))
[bench] total_ops: 7997
[bench] sa: 0  sb: 0  ss: 0  pa: 500  pb: 500
[bench] ra: 4840  rb: 1098  rr: 0  rra: 0  rrb: 1059  rrr: 0

v10.1 : Aucun mode benchmark requis.

IDENTIQUE Regles communes aux deux versions

  • Makefile avec $(NAME), all, clean, fclean, re
  • Pas de variables globales
  • 11 operations identiques (sa, sb, ss, pa, pb, ra, rb, rr, rra, rrb, rrr)
  • Instructions separees par \n
  • Sans arguments : rien afficher
  • En cas d'erreur : "Error\n" sur stderr
  • Fonctions externes : read, write, malloc, free, exit, ft_printf maison
  • Libft autorisee
  • Compilation -Wall -Wextra -Werror

Impact sur ton projet

Si tu es sur la v10.0 (projet de groupe)

Tu dois imperativement avoir :

  • CRITIQUE 4 algorithmes distincts : Simple O(n2), Medium O(n*sqrt(n)), Complex O(n log n), Adaptive
  • CRITIQUE Calcul du disorder obligatoire (tu l'as deja)
  • CRITIQUE Flags --simple, --medium, --complex, --adaptive (tu les as deja)
  • CRITIQUE Mode --bench sur stderr avec disorder%, strategie, ops detaillees (tu l'as deja)
  • CRITIQUE Complexite par regime : O(n) pour disorder <0.2, O(n*sqrt(n)) pour 0.2-0.5, O(n log n) pour ≥0.5
  • CRITIQUE README avec justification des algorithmes + contributions des 2 membres
  • CRITIQUE 2 membres dans le repo

Si tu es sur la v10.1 (projet individuel)

C'est beaucoup plus simple :

  • Pas de flags obligatoires
  • Pas de disorder obligatoire
  • Pas de benchmark obligatoire
  • Libre choix de l'algorithme, seuls les benchmarks comptent
  • README plus simple (pas de justification d'algo requise)
  • Tu peux utiliser Turk Sort partout si ca donne les meilleurs benchmarks
