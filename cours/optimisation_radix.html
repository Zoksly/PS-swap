Optimisations du Radix Sort

Pistes d'amelioration sans code tout fait -- complexite maintenue en O(n log n)

Contexte actuel

Comment fonctionne ton radix actuel

Ton radix LSD (Least Significant Bit) fonctionne ainsi :

Copier la stack dans un tableau trie pour obtenir les index de chaque element (0 a n-1)
Pour chaque bit (du bit 0 au bit max) :
Parcourir les n elements : si le bit de l'index est 0 → pb, si le bit est 1 → ra
Apres chaque passe : tout pa depuis b vers a

Cout actuel

Pour 100 nombres : les index vont de 0 a 99, donc max 7 bits.

A chaque passe de bit : n operations (ra ou pb) + nombre de pb operations (pa pour tout ramener).

Cout total approximatif : n * nb_bits + total_pa ≈ 100 * 7 + ~350 ≈ ~1050-1600 ops

Piste 1 : Skipper les passes inutiles

IMPACT ELEVE Sauter une passe quand un bit est uniforme

Si pour un bit donne, tous les elements ont la meme valeur de bit (tous 0 ou tous 1), cette passe ne fait que des ra ou que des pb suivis de pa -- elle ne change rien a l'ordre.

Principe

Avant de lancer la passe pour le bit i, compter combien d'elements ont ce bit a 0 et combien a 1. Si l'un des deux est 0, skipper toute la passe.

Sans optimisation

100 elements, 7 bits → 7 passes

Meme si le bit 6 est a 1 pour tous les elements (tous > 64), on fait quand meme 100 ra pour rien.

Avec optimisation

On detecte que le bit 6 est uniforme → skip

6 passes au lieu de 7 → economie de ~100-200 ops

Gain estime

100 nombres
-100 a -300 ops

Note : tu as deja la fonction count_bits qui fait ce calcul mais elle n'est pas utilisee pour skipper.

Piste 2 : Trier par le bit majoritaire

IMPACT MOYEN Inverser la logique pb/ra selon la majorite

Probleme

Actuellement, bit=0 → pb, bit=1 → ra. Apres la passe, tous les elements pb doivent etre pa pour revenir dans a. Si 70 elements ont le bit a 0, tu fais 70 pb + 70 pa = 140 ops rien que pour les transferts.

Principe

Compter la proportion de 0 et de 1 pour le bit courant. Envoyer dans b le groupe minoritaire (celui qui a le moins d'elements). Ainsi, le nombre de pb + pa est minimise.

Exemple : bit 3, 100 elements Cas 1 : 70 elements ont bit=0, 30 ont bit=1 Logique actuelle : pb les 70 (bit=0), ra les 30 → 70 pb + 30 ra + 70 pa = 170 ops Logique inversee : pb les 30 (bit=1), ra les 70 → 30 pb + 70 ra + 30 pa = 130 ops Economie : 40 ops sur cette seule passe

Attention

Quand tu inverses la logique (pb les 1 au lieu des 0), l'ordre de reconstruction change. Il faut reflechir a comment les elements reviennent dans a pour que le tri reste correct. Deux approches :

  • Toujours remettre les elements de b au meme endroit relatif dans a (en utilisant des pa au bon moment)
  • Alterner la direction de tri entre les passes (comme un radix LSD bidirectionnel)

Gain estime

100 nombres
-100 a -200 ops

Piste 3 : Ne pas tout ramener apres chaque passe

IMPACT ELEVE Alterner le sens du tri entre les passes (ping-pong)

Probleme

Apres chaque passe, tu fais des pa pour ramener tout de b vers a. Ces pa representent environ 30-50% des ops totales.

Principe

Au lieu de toujours trier depuis a, alterner :

Radix classique : Passe bit 0 : a → split → b contient les 0, a les 1 → pa tout → tout dans a Passe bit 1 : a → split → b contient les 0, a les 1 → pa tout → tout dans a Passe bit 2 : a → split → ... Radix ping-pong : Passe bit 0 : a → split → b contient les 0, a les 1 → on laisse comme ca Passe bit 1 : on trie depuis les deux stacks → on redistribue Passe bit 2 : ...

L'idee est de travailler avec les elements la ou ils sont, sans les deplacer inutilement. C'est plus complexe a implementer car il faut gerer les deux stacks simultanement.

Variante simplifiee

Alterner le "sens" : passe impaire trie de a vers b, passe paire trie de b vers a. Ainsi, les pa de reconstruction d'une passe deviennent les pb de tri de la passe suivante -- ils ne sont plus "gratuits" mais il n'y a plus de phase de reconstruction separee.

Gain estime

100 nombres
-200 a -400 ops

Piste 4 : Utiliser un radix en base plus grande

IMPACT MOYEN Radix en base 4 au lieu de base 2

Probleme

En base 2, pour 100 elements tu as 7 passes. Chaque passe coute ~n ops minimum.

Principe

En base 4, chaque "digit" encode 2 bits. Tu as besoin de ceil(7/2) = 4 passes au lieu de 7. Mais chaque passe doit separer les elements en 4 groupes (digits 0, 1, 2, 3).

Avec 2 stacks, tu ne peux separer qu'en 2 groupes a la fois. Mais tu peux faire 2 sous-passes :

Base 4, digit courant peut etre 0, 1, 2, ou 3 : Sous-passe A : envoie les digits 0 et 1 dans b (bit de poids fort du digit = 0) garde les digits 2 et 3 dans a Sous-passe B : dans b, separe les 0 des 1 (ra/pb) dans a, separe les 2 des 3 (ra/pb) Resultat : les 4 groupes sont ordonnes en 2 sous-passes au lieu de 2 passes completes

Difficulte

La gestion de la reconstruction est plus delicate. Il faut recombiner les 4 groupes dans le bon ordre, ce qui demande une logique plus sophistiquee que le radix binaire.

Gain estime

100 nombres
-150 a -300 ops

Piste 5 : Pre-tri partiel avant le radix

IMPACT MOYEN Reduire le nombre de bits necessaires

Principe

Avant de lancer le radix, faire un pre-tri "grossier" qui place les elements approximativement dans le bon ordre. Si la stack est deja partiellement triee, les bits de poids fort seront uniformes et pourront etre skippes (voir Piste 1).

Exemple avec 100 elements (index 0-99) : Sans pre-tri : index melangees, 7 bits necessaires, aucun bit skippable Avec pre-tri (push les moities) : 1. Push les elements dont l'index < 50 dans b (~150 ops) 2. Ramene-les dans a dans un ordre approximatif (~150 ops) 3. Maintenant les bits de poids fort sont quasi uniformes par zones 4. Le radix demarre avec moins de passes effectives Cout du pre-tri : ~200-300 ops Economie sur le radix : 1-2 passes en moins = ~200-400 ops Bilan net : -0 a -200 ops

Attention

Le pre-tri ajoute des operations. Le gain net depend de la qualite du pre-tri. Si mal calibre, ca peut augmenter le total.

Piste 6 : Optimiser les pa de fin de passe

IMPACT FAIBLE Exploiter l'ordre dans b avant de pa

Principe

Apres une passe de radix, les elements dans b sont dans un certain ordre. Si les premiers elements de b sont deja proches de leur position finale dans a, on peut les pa de maniere strategique plutot que tout d'un coup.

Par exemple, si apres la passe du bit 2, les elements dans b sont partiellement tries pour les bits 0-1 (grace aux passes precedentes), certains sont deja bien places.

Gain estime

100 nombres
-20 a -80 ops

Piste 7 : Commencer par le MSB (Most Significant Bit)

IMPACT MOYEN Radix MSD au lieu de LSD

Principe

Ton radix actuel est LSD (part du bit 0). Un radix MSD part du bit le plus significatif et descend.

Avantage

Le MSD cree des partitions des le depart. Apres la premiere passe (bit de poids fort), tu as deux groupes : les "petits" (bit=0) et les "grands" (bit=1). Chaque groupe est deja dans la bonne moitie.

Pour les passes suivantes, tu ne tries que les elements qui en ont besoin. Si un sous-groupe est deja trie, tu le skippes.

Radix MSD sur 8 elements (index 0-7, 3 bits) : Passe bit 2 : separe {0,1,2,3} et {4,5,6,7} → 2 groupes Passe bit 1 : dans chaque groupe, separe les paires → 4 groupes Passe bit 0 : dans chaque paire, swap si necessaire → trie Avantage : si {4,5,6,7} est deja trie apres la passe bit 2, les passes bit 1 et 0 ne s'appliquent qu'a {0,1,2,3}

Difficulte

Avec seulement 2 stacks, gerer les sous-groupes recursifs est complexe. Il faut garder en memoire les bornes de chaque partition. Faisable mais plus difficile a implementer que le LSD.

Gain estime

100 nombres
-100 a -300 ops

Verdict honnete : meme avec toutes les optimisations, descendre sous 700 ops pour 100 nombres avec un radix O(n log n) est extremement difficile. Le plancher theorique d'un radix pour 100 nombres est autour de ~600-700 ops, ce qui laisse tres peu de marge.

La piste 1 (skip des passes uniformes) est la plus facile et la plus rentable en ratio effort/gain. C'est celle a implementer en premier.

Rappel important

Le seuil "excellent" (< 700 ops pour 100 nombres) est un objectif global de performance, teste probablement avec le mode --adaptive. Si les inputs random tombent souvent dans le regime high disorder (≥ 0.5), tu seras force en O(n log n), et ~700-1000 ops optimise est le mieux que tu puisses esperer.

Le palier "bon" (< 1500) est largement atteignable avec les pistes 1 et 2 seules.
