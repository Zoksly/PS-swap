2. Bugs critiques

CRITIQUE BUG #1 — Pas de vérification "déjà trié"

main.c:122-123

Après calcul du disorder, ft_choose_algo est appelé même si la liste est déjà triée. Aucun if (is_sorted_stack(...)) ni if (disorder == 0) return.

tab -> disorder = ft_compute_disorder(tab, tab -> lena);
// MANQUE : if (is_sorted_stack(tab->stack_a)) { ft_free(args, tab); return 0; }
ft_choose_algo(tab);

Conséquence : ./push_swap 1 2 affiche sa (swap la liste déjà triée → résultat FAUX).

CRITIQUE BUG #2 — small_sorters swap inconditionnel pour 2 nombres

Algos/small_sorters.c:21-22

else if ((int)tab -> lena == 2)
    sa(tab, &tab -> stack_a);  // TOUJOURS swap, meme si deja trie !

Fix : Ajouter if (!is_sorted_stack(tab->stack_a)) avant le sa.

CRITIQUE BUG #3 — Comparaison void* au lieu de (long) dans ft_compute_disorder

main.c:31-34

if ((current -> content) > next -> content)      // void* comparison !
    miss++;
else if ((current -> content) < next -> content)  // void* comparison !
    succ++;

Les nombres négatifs sont stockés comme (void*)(long)(-5) = 0xFFFF...FFFB. En comparaison non-signée, -5 > 4 est VRAI.

Conséquence : -5 -4 -3 ... 4 (trié) donne disorder > 0 → 46 opérations inutiles.

Fix : if ((long)(current->content) > (long)(next->content))

CRITIQUE BUG #4 — Fuite mémoire : ft_lstclear avec del=NULL

utils.c:20-23 + libft/ft_lstclear.c

// utils.c
ft_lstclear(&tab -> stack_a, NULL);  // del = NULL
ft_lstclear(&tab -> stack_b, NULL);

// libft/ft_lstclear.c
void ft_lstclear(t_list **lst, void (*del)(void*))
{
    if (lst && del)  // <-- del est NULL, donc RIEN n'est libere !
    { ... }
}

Conséquence : 80+ bytes de fuite à chaque exécution (tous les noeuds de la liste).

Fix : Soit modifier ft_lstclear, soit libérer manuellement dans ft_free :

t_list *tmp;
while (tab->stack_a) {
    tmp = tab->stack_a->next;
    free(tab->stack_a);
    tab->stack_a = tmp;
}
CRITIQUE BUG #5 — ft_dprintf ne supporte pas %f

push_swap.c:68

ft_dprintf(2, "[bench] Disorder : %.2f%%\n", (int)(disorder));

Problèmes : (1) ft_dprintf n'implémente pas %f/%.2f, (2) cast en (int) avant passage à un format float.

Fix : Utiliser ft_dprintf(2, "[bench] Disorder : %d%%\n", (int)(disorder));

CRITIQUE BUG #6 — Lecture hors bornes dans find_chunk_in_stack

Algos/medium.c:30

index = get_index(stack, sorted_array, tab->lena + tab->lenb);
//                                      ^^^^^^^^^^^^^^^^^^^^
// sorted_array est alloué avec taille tab->lena (ligne 70 de ft_medium)
// mais on cherche sur tab->lena + tab->lenb elements !

Conséquence : Lecture hors du buffer alloué dès que lenb > 0. UB potentiel.

CRITIQUE BUG #7 — Mode Adaptive utilise le radix (~11000 ops) au lieu du Turk (~5000 ops)

push_swap.c:15-25

void adaptative_mode(t_sta *tab)
{
    if (tab->lena >= 2 && tab->lena <= 5)
        small_sorters(tab);
    else if (tab->disorder >= 0.5)
        radix_sort(tab);        // 7500-11800 ops pour 500 nombres !
    else if (tab->disorder >= 0.2)
        ft_medium(tab, ...);
    else if (tab->disorder < 0.2)
        ft_simple(tab);         // Selection sort O(n^2)
}

Le mode par défaut (ADAPTIVE) ne passe jamais les limites de la moulinette pour 100+ nombres. Le Turk sort (--simple) donne ~4900 ops pour 500 nombres.

5. Performance & optimisations

PERF Le mode Adaptive devrait utiliser Turk sort pour les grandes listes

Actuellement, pour disorder ≥ 0.5 (cas le plus fréquent avec des données aléatoires), le radix sort est utilisé :

Algo500 nombres (random)500 nombres (inversé)Objectif
Radix (adaptive)7500-118006784< 5500
Turk (--simple)~4900~5100< 5500
Medium (chunks)~6000~5800< 5500

Suggestion : Utiliser le Turk sort comme algo principal pour ≥ 6 nombres dans le mode adaptive.

PERF sort_array utilise bubble sort

helpers.c:42-65 — Tri à bulles O(n²). Pour 500 nombres, c'est lent. Un tri par insertion ou merge sort serait plus rapide. Impact mineur sur le nombre d'opérations push_swap mais ralentit l'exécution.

PERF ft_chunk_size : trous dans les conditions

helpers.c:15-25

if (i > 20 && i < 150)     // i=20, i=150 non couverts
else if (i > 150 && i < 300) // i=300 non couvert
else if (i > 300)
else return (i / 2);         // i=20 tombe ici

Les valeurs 20, 150, et 300 tombent dans le else (i/2). Utiliser >= et <=.

PERF ft_simple : selection sort O(n²) pour le mode adaptive faible disorder

Algos/simple.c:39-61 — Utilisé quand disorder < 0.2. Même pour peu de désordre, c'est très lent sur 500+ nombres. Le Turk sort serait meilleur dans tous les cas.

PERF get_index : recherche linéaire dans le tableau trié

helpers.c:27-39 — Recherche O(n) dans un tableau trié. Une recherche binaire serait O(log n). Impact mineur.

6. Problèmes divers

ROBUSTESSE Guard incomplet dans ft_r (op3.c)

Operations/op3.c:20

if (!(stack1) || !((*stack1)->next))
//                 ^^^^^^^^^^^^^^^^ dereference *stack1 sans verifier si *stack1 est NULL

Si *stack1 == NULL, (*stack1)->next est un segfault. Devrait être :

if (!(stack1) || !(*stack1) || !((*stack1)->next))

Note : op4.c a déjà le guard correct.

TYPE Retour 0 au lieu de NULL pour int*

helpers.c:75 : return (0); dans stack_to_array qui retourne int *. Devrait être return (NULL);

TYPE Troncature long → int dans stack_to_array

helpers.c:79 : array[i] = (long)stack->content;

Le tableau est int* mais on y stocke un long. Pour des valeurs dans la plage INT, ca fonctionne, mais c'est un cast implicite dangereux.

ft_dprintf.h:31 re-déclare ft_strlen déjà dans libft.h.

STYLE Le fichier small_sorters.c a pour header hardcodes.c
